计算机网络第二章学习笔记

发表于 2020-01-07 更新于 2020-11-23 分类于 课程

应用层

应用层很重要。为什么重要？因为如果我们不能构想出任何有用的应用，也就没有必要去设计支持它们的网络协议了。

应用层协议原理

网络应用程序体系结构

客户-服务器体系结构(client-server architecture)

在这个体系结构中，有一个总是打开的主机称为服务器，它服务于来自许多其他称为客户的主机的请求。如apache就是一个非常著名的Web服务器。这种体系结构中，客户之间不相互直接通信。主机拥有一个固定的IP地址，客户就可以通过向这个固定的IP地址发送分组来与其联系。
采用了客户-服务器体系结构的应用程序包括：

  • Web
  • FTP
  • Telnet
  • 电子邮件

P2P体系结构(P2P architecture)

在P2P体系结构中，对于服务器的以来没有那么大（或者直接没有），相反，应用程序在间断连接的主机对之间使用直接通信。这些主机对被称为对等方。目前须有流行的、流量密集型的应用都是通过P2P体系结构的。包括：

  • 文件共享(BitTorrent)
  • 对等方协助下载加速器(迅雷)
  • 因特网电话(skype)
  • IPTV(PPstream)

P2P体系结构最引人入胜(NB)的特性之一是它的自扩展性(self-scalability)。例如，在一个P2P文件共享型应用中，景观每个对等方都由于请求文件产生工作量，但每个对等方通过向其他对等方发送文件也为系统郑家了服务能力（人人为我，我为人人）。P2P同时也是成本有效的（省钱）。P2P应用面临的三个主要挑战：

  • ISP友好。接入ISP的一般受限于非对称的带宽应用，下载要比上载多得多。所以需要一个对ISP更友好的模式。
  • 安全性。它的高度分布和开放性给安全带来了巨大挑战。
  • 激励。说服用户提供带宽、存储和计算资源。

进程通信

网络间的进程通信和操作系统中的进程通信是有区别的，这里我们更关注端系统到端系统之间的进程进行通信。
在两个不同的端系统中，通过跨越计算机网络交换**报文(message)**而相互通信。

客户和服务器进程

网络应用程序由成对的应用组成。这些进程之间相互发送报文。对于每对通信，我们通常将这两个进程之一标识为客户(client)**，而另一个进程标识为服务器(server)。通常来说发起通信的被标识为客户，而等待联系**的是服务器。

进程与计算机网络之间的接口

进程通过套接字（socket）的软件接口向网络发送接收报文。把进程比作一座房子，那么套接字就是它的门，它通过这个门把报文送出去。
套接字是同一台主机内应用层与运输层之间的接口，也称为应用程序和网络之间的**应用程序编程接口(Application Programming Interface)**。应用程序的开发者能且只能控制套接字在应用层端的一切，对运输层端几乎没有控制权。应用程序的开发者对应用层的控制仅限于：

  1. 选择运输层协议。
  2. 也许能设定几个运输层参数，如最大缓存和最大报文长度。

    进程寻址

    不同端系统之间的进程要想通信，需要明确：
  3. 主机的地址。
  4. 定义在目的主机中的接收进程标识符。

在因特网中，主机由其IP地址(IP address)**标识。接收进程用端口号(port number)**指定。

可供应用程序使用的运输服务

根据应用程序的不同服务要求，我们大体可以从这四个方面对要求的运输层服务进行分类：

  1. 可靠数据传输
  2. 吞吐量
  3. 定时
  4. 安全

可靠数据传输

第一章中谈到了分组的丢失(lost)，然而在一些情况下我们必须保证数据正确、完成的交付给应用程序的另一端。如果一个协议提供了这样的确保数据交付的服务，就认为提供了**可靠数据传输(reliable data transfer)**。这时在应用层编程的我们只需要将数据传给套接字就可以完全相信数据会无误地传给接收进程。

吞吐量

越大越好。

定时

耍游戏的时候延迟太高了影响游戏体验。运输层协议也能提供定时保证。（这么牛皮？）

安全性

运输协议能够为应用程序提供一种或多种安全性服务。如加密所有数据等。

因特网提供的运输服务

因特网（更一般的是TCP/IP网络）为应用程序提供两个运输成协议，UDP和TCP。

TCP服务

TCP服务模型包括面向连接服务和可靠数据传输服务。当某个应用程序调用TCP作为它的运输协议时，这个应用程序就获得了来自TCP这两种服务：

  1. 面向连接的服务：在应用层数据报文开始流动之前，TCP让客户和服务器相互交换运输层控制信息。这个所谓的握手过程提示客户和服务器，使他们为大量分组的到来做好准备。在握手阶段后，一个**TCP连接(TCP connection)**就可以在两个套接字之间建立。这个连接是全双工的（连接的双方都可以在这个连接上同时进行报文的收发）。应用程序结束报文的发送时，必须拆除该连接。
  2. 可靠的数据传送服务：通信进程可以依赖TCP进行无差错、按适当顺序交付所有发送的数据。应用程序一端将字节流传入套接字时，它可以依靠TCP将相同的字节流家父给接收方的套接字，并且没有字节的丢失和冗余。

TCP还具有拥塞控制机制。它不一定能为通信进程带来好处，但是它对因特网整体来说是有益的。

UDP服务

UDP是一种轻量级的运输协议，它提供最小服务。UDP的特点如下：

  • 无连接
  • 不可靠

TCP和UDP的安全性

无论是TCP还是UDP都没有提供任何加密机制。
对于TCP，已经有**安全套接字层（Secure Sockets Layer, SSL)**，用SSL加强的TCP不仅能够做到传统的TCP可以做到的一切，并且提供关键的进程到进程的安全性服务，包括：

  • 加密
  • 数据完整性
  • 端点鉴别

但是SSL并不是TCP和UDP在相同层次上的第三种因特网运输协议（也就是说SSL并不是一种因特网运输层的运输协议），而是对TCP的一种加强，并且这种加强是在应用层上实现的。

应用层协议

**应用层协议(application layer protocol)**定义了运行在不同端系统上的应用程序进程如何相互传递报文。它定义了：

  • 交换的报文类型，例如请求报文和响应报文。
  • 各种报文类型的语法，如报文中的各个字段以及这些字段是如何描述的。
  • 字段的语义，即这些字段中包含信息的含义。
  • 一个进程何时以及如何发送报文，对报文的响应规则。

有些应用层协议由RFC文档定义的，因此他们位于公共域中。例如Web应用层协议HTTP([RFC2616])。如果浏览器开发者村从HTTP RFC规则，所开发出的浏览器就能访问任何遵从该文档标准的Web服务器并获取相应的Web页面。有的应用层协议是专用的，有意不为公共域使用。

Web和HTTP

HTTP概况

Web的应用层协议是超文本传输协议（HyperText Transfer Protocol），它是Web的核心。定义在[RFC 1945]和[RFC 2616]中。HTTP由两个程序实现。一个客户端程序，一个服务器程序。他们运行在不同的端系统中，通过交换HTTP报文进行会话。
Web术语：**Web页面(Web page)**（也叫文档）是由对象组成的。一个对象(object)只是一个文件，如一个HTML文件、一个JPEG图形、一个Java小程序或者一个视频片段。且他们可以通过一个URL地址寻址。多数Web页面含有一个HTML基本文件(base HTML file)以及几个引用对象。例如：一个Web页面包含HTML文本和5个JPEG图形，那么这个Web页面有六个对象：一个HTML基本文件加5个图形。Web浏览器(Web browser)实现了HTTP客户端，Web服务器(Web server)实现了HTTP的服务器端，用于存储Web对象，对每个对象用URL寻址。
HTTP定义了Web客户向Web服务器请求Web页面的方式，以及服务器向客户传送Web页面的方式。
HTTP使用TCP作为它支撑运输的协议。HTTP客户首先发起一个与服务器的TCP连接。一旦连接建立，该浏览器和服务器进程就可以通过套接字接口访问TCP。
HTTP是一个无状态协议(stateless protocol)。也就是说在某个时刻客户请求了一个对象，在下一次请求这个对象的时候服务器不会因为刚才已经为客户提供了这个对象而不再响应这个请求，而是重新发送这个对象。

非持续连接和持续连接

  1. 非持续连接下的请求Web页面的步骤：

    1. HTTP客户进程在端口号80（HTTP默认端口号）发起一个到服务器例如Hooolin.github.io的TCP连接，在客户和服务器上分别有一个套接字与该连接相关联。
    2. HTTP客户通过它的套接字想服务器发送一个HTTP请求报文。请求报文中包含了/路径名/index.html。
    3. HTTP服务器进程经它的套接字接收该请求报文，从存储器中检索出对象Hooollin.github.io/index.html，在一个HTTP响应报文中封装对象，并通过其套接字向客户发送响应报文。
    4. HTTP服务器通过TCP断开该TCP连接。（实际上是TCP确认客户已经完整的收到报文为止，它才会实际中断连接）。
    5. HTTP客户接收响应报文，TCP连接关闭。报文指出封装的对象是一个HTML文件，客户从响应报文中提取该文件，检查该HTML，得到对10个JPEG图形的引用。
    6. 对每个引用的JPEG图形对象重复前4个步骤。

    从上面的步骤可以看到，每个TCP连接都是在服务器发送一个对象之后关闭，并没有为其他的对象而持续下来。值得注意的是每个TCP连接只传输一个请求报文和一个响应报文。本例中用户请求该Web网页时，要产生11个TCP连接。

    **往返时间(Round-Trip Time, RTT)**：指一个短分组从客户到服务器然后再返回客户所花费的时间。RRT包括：

    • 分组传播时延
    • 排队时延
    • 处理时延

    那么简单估算一下从客户请求HTML基本文件到客户收到整个文件所花费的时间。当用户点击超链接时，会英气浏览器和Web服务器之间发起一个TCP连接，涉及一个三次握手的过程。即：

    1. 客户向服务器发送一个小TCP报文段。
    2. 服务器用一个小TCP报文段做出确认和响应。
    3. 最后，客户向服务器返回确认。


    从2-7可以看出，是三次握手中的前两个部分用去了一个RTT。第三部分向TCP连接发送一个HTTP请求报文，一旦请求报文达到服务器，服务器就在TCP连接上发送HTML文件。该HTTP请求/响应用了另一个RTT。粗略的说，总的响应时间就是两个RTT加上服务器传输HTML文件的时间。

  2. 采用持续连接的HTTP
    非持续连接有一些缺点：

    • 必须为每一个请求的对象建立和维护一个全新的连接。这样给Web服务器带来了严重的负担。

    • 每一个对象经受两倍的RTT的交付延迟。

      在采用持续连接的情况下，服务器在发送响应后保持该TCP连接打开。在相同的客户与服务器之间的后续请求和响应报文能够通过相同的连接进行传送。

HTTP报文格式

HTTP请求报文

典型的HTTP请求报文：

HTTP请求报文的第一行叫做请求行(request line)**，其后继的行叫做首部行(header line)。请求行有三个字段：方法字段、URL字段和HTTP版本字段。方法字段可以取GET、POST、HEAD、PUT和DELETE。
首部行的作用很显然。

从通用格式中可以看到，在首部行后一个一个实体体(entity body)**。使用GET方法时实体体为空，使用POST方法时才使用该实体体。

HTTP响应报文

典型的HTTP响应报文：


响应报文分为三个部分：

  • **状态行(status line)**。包括三个字段：协议版本字段、状态码字段和响应状态信息字段。
  • **首部行(header line)**。对象类型应该正式地由Content-Type指出而不是用文件拓展名来指示。
  • **实体体(entity body)**。

常见的状态码和状态短语包括：

  • 200 OK:请求成功，信息在返回的响应报文中。
  • 301 Moved Permanently:请求的对象已经被永久转移了，心的URL定义在响应报文中的Location:首部行中。客户软件自动获取心的URL。
  • 400 Bad Request:一个通用差错代码，指示该请求不能被服务器理解。
  • 404 Not Found:被请求的文档不再服务器上。
  • 505 HTTP Version Not Supported:服务器不支持请求报文使用的HTTP协议版本。

用户与服务器的交互：cookie

HTTP是无状态的。但是一个Web站点其实希望能够识别用户，将用户身份与内容联系起来。为此，HTTP使用了cookie。

cookie技术有四个组件：

  1. 在HTTP响应报文中的一个cookie首部行；
  2. 在HTTP请求报文中的一个cookie首部行；
  3. 在用户端系统中保留有一个cookie文件，并由用户的浏览器进行管理；
  4. 位于Web站点的一个后端数据库。

Web缓存

Web缓存器(Web cache)**也叫做代理服务器(proxy server)**，它是能够代表初始Web服务器来满足HTTP请求的网络实体。
Web缓存器中有自己的磁盘存储空间，并在存储空间中保存最近请求过的对象副本。可以通过配置客户的浏览器，使得用户的所有HTTP请求首先指向Web缓存器。举例来说，假设客户正在请求一个对象，将会发生如下情况：

  1. 浏览器建立一个到Web缓存器的TCP连接，并向Web缓存器中的对象发送一个HTTP请求。
  2. Web缓存器进行检查，看看本地是否存储了该对象副本。如果有，Web存储器用HTTP响应报文返回该对象。
  3. 如果Web存储器中没有该对象，他就打开一个与该对象的初始服务器的TCP连接，并发送一个对该对象的HTTP请求。初始服务器收到这个请求后，向Web缓存器发送具有该对象的HTTP的响应。
  4. 当Web缓存器接收到了该对象后，将在本地存储空间中存储一份副本，并向客户的浏览器用HTTP响应报文发送该副本。

在因特网上部署Web服务器有两个原因：

  • Web缓存器可以大大减少对客户请求的响应时间。特别是当客户与初始服务器之间的瓶颈带宽远低于客户与Web缓存器之间的瓶颈带宽时更是如此。
  • Web缓存器能够大大减少一个机构的接入链路到因特网的通信量。比如说我们学校配置一台Web缓存器供全校使用，在校园局域网内链路速度是100Mbps，而接入公共英特网的链路只有15Mbps的速度，那么我们通过访问Web缓存器获取对象的速度要比通过访问初始服务器的速度快得多。（涉及到命中以及未命中的情况）

**内容分发网络(Content Distribution Network, CDN)**，使得Web缓存器的作用越来越重要。

条件GET方法

尽管高速缓存能够减少用户感受到的响应时间，但也引入了一个显而易见的问题：存放在缓存器中的对象副本可能是陈旧的（原对象已被修改）。
HTTP协议中的**条件GET(conditional GET)**可以解决这个问题。如果请求报文中：

  1. 使用GET方法；
  2. 包含一个”If-Modified-Since”首部行

那么这个HTTP请求报文就是一个条件GET请求报文。
例如一个代理缓存器访问初始服务器获取对象并通过条件GET更新对象的过程大致如下：

  1. 代理缓存器向某Web服务器发送一个HTTP请求报文。
  2. Web服务器向缓存器发送具有被请求对象的响应报文。并且返回的响应报文中包含”Last-Modified”的首部行。
  3. 代理缓存器缓存该对象，同时也存储了最后修改日期。
  4. 一段时间后，代理服务器向Web服务器发送一个条件GET执行最新检查。
  5. Web服务器返回一个响应报文，状态行为HTTP/1.1 304 NOT Modified。
  6. 代理缓存器接收到响应报文，并确定对象副本并没有被修改。

文件传输协议：FTP

FTP使用了两个并行的TCP连接来传输文件，一个是控制连接(control connection)**（TCP控制连接端口21），一个是数据连接(data connection)。（TCP数据连接端口20）控制连接用于在量主机之间阐述控制信息，如用户标识、口令、改变远程目录的命令以及“存放(put)”和“获取(get)”文件的命令。数据连接用于实际发送一个文件。因为一个FTP协议使用一个独立的控制连接，所以我们也称FTP的控制信息是带外(out-of-band)传送的。HTTP协议事在传输文件的同一个TCP连接中发送请求和响应首部行的，因此HTTP也可以说是带内(in-band)发送控制信息的。
FTP服务器必须在整个会话期间保留用户的状态(state)**。

因特网中的电子邮件

因特网中电子邮件组成部分：

  • **用户代理(user agent)**：用户代理允许用户阅读、回复、转发、保存和撰写报文。
  • 邮件服务器(mail server)**：电子邮件体系的核心。每个接收方在某个邮件服务器上有一个邮箱(mailbox)**。邮箱管理和维护着发送给他的报文。
  • 简单邮件传输协议(SMTP)

一个典型的邮件发送过程：从发送方的用户代理开始，传输到发送方的邮件服务器，再传输到接收方的邮件服务器，然后在这里被分发到接收方的邮箱中。

DNS：因特网的目录服务

主机可以有多种表示形式，其中一种方式是通过主机名(hostname)**来标识。比如说baidu.com、bilibili.com等。这种表示方式对人较为友好，但是它几乎没有提供什么有用的信息，并且主机名可能由不定长的字母数字组成，路由器很难处理。因此，主机也可用IP地址(IP address)**进行表示。

DNS提供的服务

人们便于记忆主机名地标识方式，但是路由器更喜欢定长的、有结构层次的IP地址。那么一种折中的方法就是**域名系统(Domain Name System, DNS)**。
DNS是：

  1. 一个由分层的**DNS服务器(DNS server)**实现的分布式数据库。
  2. 一个使得主机能够查询分布式数据库的应用层协议。

DNS服务器通常是运行BIND(Berkeley Internet Name Domain)软件的UNIX机器，运行在UDP之上，使用53号端口。
DNS通常是由其他应用层协议所使用的，包括HTTP、SMTP、FTP，将用户提供的主机名解析为IP地址。
一个典型的地址转换过程如下：

  1. 用户主机上运行着DNS应用的客户端。
  2. 浏览器从用户访问的URL中抽取出主机名，并将这台主机名传给DNS应用的客户端。
  3. DNS客户向DNS服务器发送一个包含主机名的请求。
  4. DNS客户最终会收到一份回答报文，其中含有对应于该主机名的IP地址。
  5. 一旦浏览器接收到来自DNS的该IP地址，它能够向位于该IP地址80端口的HTTP服务器进程发起一个TCP连接。

除了进行主机名到IP地址的转换，DNS还提供一些重要的服务：

  • **主机别名(host aliasing)**。有复杂主机名的主机可能有一个或者多个别名。通常来说主机别名比规范主机名更加好记一些。
  • **邮件服务器别名(cannonical hostname)**。
  • 负载分配(load distribution)**。繁忙的站点被冗余分布在多台服务器上，每台服务器都运行在不同的端系统上，每个都有着不同的IP地址。一个IP地址集合**与同一个规范主机名相联系。

DNS工作机理概述

DNS的一种简单设计（最容易想到）就是在全球范围内搞一个DNS服务器，这个服务器包含所有的主机名和IP地址的映射。这种集中式的设计不适用于当今的因特网，是因为因特网有着数量庞大的主机。除此之外还有一些设计问题，包括：

  • 单点故障(a single point of failure)。如果DNS服务器崩溃，整个因特网随之崩溃。
  • 通信容量(traffic volume)。单个DNS服务器要处理全球所有的DNS查询。
  • 远距离的集中式数据库(distant centralized database)。很显然DNS服务器不可能临近所有查询的客户。这将导致很严重的时延。
  • 维护。单个DNS服务器不得不为所有的因特网主机保留记录。所以这个中央数据库会非常庞大，难以维护。

总的来说，单一DNS服务器上运行集中式数据库完全没有可拓展能力。所以DNS采用了分布式的设计方案。

分布式、层次数据库

为了处理拓展性问题，DNS使用了大量的DNS服务器，以层次方式组织，并且分布在全世界范围内。没有一台DNS服务器拥有因特网上所有主机的映射。

有三种类型的DNS服务器：

  • 根DNS服务器。根服务器位于层次结构的最顶层。根服务器中的每个都可以视为单个的服务器，但每台服务器实际上是一个冗余服务器的网络，以提供安全性和可靠性。
  • 顶级域(Top-Level Domain, TLD)DNS服务器。这些服务器负责顶级域名如com、org、net、edu和gov,以及所有国家的顶级域名如uk、fr、和ca。
  • 权威DNS服务器。在因特网上具有公共可访问主机的每个组织机构必须提供公共可访问的DNS记录，这些记录将这些主机的名字映射为IP地址。一个组织机构的权威DNS服务器收藏了这些DNS记录。（比如说阿里云里面注册一个域名，他就相当于一个权威DNS服务器的提供商）。

除了根、TLD和权威DNS服务器之外，还有一类重要的DNS，称为**本地DNS服务器(local DNS server)**。本地DNS服务器骑着代理的作用，并将该请求转发到DNS服务器层次结构中。

看一个简单的查询过程的例子。假设主机cis.poly.edu想知道主机gaia.cs.umass.edu的IP地址。同时假设理工大学(Polytechnic)的本地DNS服务器为dns.poly.edu，并且gaia.cs.umass.edu的权威DNS服务器为dns.umass.edu。

  1. 主机cis.poly.edu首先向他的本地DNS服务器dns.poly.edu发送一个DNS查询报文。该查询报文含有被转换的主机名 gaia.cs.umass.edu。
  2. 本地DNS服务器将该报文转发到根DNS服务器。
  3. 根DNS服务器注意到其edu的前缀并向本地DNS然会负责edu的TLD的IP地址列表。
  4. 该本地DNS服务器则向TLD发送一个查询报文。
  5. TLD服务器注意到 umass.edu的前缀，并用权威DNS服务器的IP地址进行响应，返回负责马萨诸塞大学的权威DNS服务器 dns.umass.edu。
  6. 最后，本地DNS服务器直接向 dns.umass.edu重发查询报文，dns.umass.edu用 gaia.cs.umass.edu的IP地址进行响应。


整个过程一共发送了四份查询报文和四分回答报文。开销较大。

这个例子中利用了递归查询(recursive query)**和迭代查询(iterative query)**。通常来说，主机到本地DNS服务器的查询通常是递归的，其余的查询通常是迭代的。

DNS缓存

DNS缓存技术非常简单，但是运用十分广泛。简而言之就是DNS服务器可以缓存主机到IP地址的映射，但是会在一段时间后丢弃缓存信息。

DNS记录和报文

共同实现DNS分布式数据库的所有DNS服务器存储了**资源记录(Resource Record, RR)**，RR提供了主机名到IP地址的映射。每个DNS回答报文包含了一条或者多条资源记录。
资源记录是一个包含了下列字段的四元组：(Name, Value, Type, TTL)。
TTL是该记录的生存时间，他决定了资源记录应当从缓存中删除的时间。Name和Value的值取决于Type：

  • 如果Type = A，则Name是主机名，Value是该主机名对应的IP地址。一条类型为A的资源记录提供了标准的主机名到IP地址的映射。
  • 如果Type = NS，则Name是个域（如foo.com)，而Value是个知道如何获得该域中主机IP地址的权威DNS服务器的主机名。这个记录用于沿着查询链来路由DNS查询。
  • 如果Type = CNAME, 那么Value是别名为Name的主机对应的规范主机名。这个记录能过向查询主机提供一个主机名对应的规范主机名。
  • 如果Type = MX，则Value是个别名为Name的邮件服务器的规范主机名。

DNS报文

DNS查询报文和回答报文有着相同的格式。

  • 前12个字节是首部区域，包括几个字段。第一个字段是一个16比特的数，用来标识该查询。这个标识符会被复制到对查询的回答报文中，以便让客户用它来匹配发送的请求和接收到的回答。
  • 问题区域包括正在进行的查询信息。该区域包括：
    • 名字字段，指出被查询主机的名字；
    • 类型字段，指出有关该名字的正被询问的问题类型；
  • 回答区域包含了对最初请求的名字的资源记录。回答报文的回答区域中可以包含多条RR，因此一个主机名可能有多个对应IP地址（冗余服务器）。
  • 权威区域包含了其他权威服务器的记录。
  • 附加区域包含了其他有帮助的信息。

在DNS数据库中插入记录

比如说我创建了一个教育机构叫做HollinEducation，那么我需要去注册登记机构上注册域名hollineducation.edu。注册登记机构(registrar)是一个商业实体，它验证该域名的唯一性，将该域名输入DNS数据库，并收取少量的费用。
当我注册域名的时候，我需要向该机构提供我的基本和辅助权威DNS服务器的名字和IP地址。假定该名字和IP地址是dns1.hollineducation.com以及212.212.212.1。该注册登记机构将我的资源记录插入到DNS系统中。

P2P应用

P2P文件分发

现在最常见的文件下载方式就是从单一的服务器向大量的主机（称为对等方）分发一个大文件。在这种情况下，服务器必须向每个对等方发送一个文件的副本，这样就会占用大量的服务器带宽，使得服务器承受极大的压力。
在P2P文件分发中，每个对等方能够重新分发它所有的该文件的任何部分，从而在分发的过程中协助该服务器。现在最为流行的P2P文件共享协议是BitTorrent。

P2P体系结构的拓展性

对于客户-服务器体系结构的分发时间可以很容易计算出来：

  • 服务器对于每个客户都需要传送一份文件，那么总的分发文件长度为$NF$，分发该文件的时间为$NF/u$。
  • 每个客户接收文件的速度都不同，那么最长的接收时间就是$F/d_{min}$。

那么最后的结果就是
$$D_{cs} \geq max{( \frac{NF}{u_s},;\frac{F}{d_{min}} )}$$
对于P2P体系的体系结构分发时间进行分析可以得到：

  • 首先只有服务器有要分发的文件，服务器至少要将这个文件的每个bit都经过其接入链路。所以最小分发时间至少是$\frac{F}{u}$。
  • 现在考虑每个对等方，他们都不能以小于$\frac{F}{d_{min}}$秒的分发时间获得所有的比特。
  • 系统整体的总上载能力等于服务器上载速率加上每个单独的对等放的上载速度。$u_{total}=u_s+u_1+u2+…+u_N$。系统必须向这N个对等方的每个交付F比特，所以总共交付$NF$比特。这不能以快于$u_{total}$的速率完成。所以最小分发时间至少是$\frac{NF}{u_s+u_1+…+u_N}$。

BitTorrent

BitTorrent是用于一种文件分发的P2P协议。参与一个特定文件分发的所有对等方的集合被称为一个洪流(torrent)**。在一个洪流中的对等方下载等长度的文件块(chunk)，典型的块长度为256KB。当一个对等方首次加入一个洪流时，它没有块。随着时间的流逝，它积累了越来越多的块。当它下载块时，也为其他对等方上载了多个块。一旦某个对等方获得了整个文件，那么它也许会离开洪流，或留在该洪流中并继续向其他对等方上载块。
BitTorrent运行的过程大致如下：每个洪流具有一个基础设施节点，称为追踪器(tracker)**。当一个对等方加入某洪流时，他向追踪器注册自己，并周期性地通知追踪器它仍在洪流中。以这种方式，追踪器追踪正参与在洪流中的对等方。一个给定的洪流可能在任何时候具有数以千计的对等方。

分布式散列表

考虑如何通过一个P2P网络实现一个简单的分布式数据库。那么在P2P网络中的分布式数据库，每个对等方只保持总体的一个子集。允许任何一对对等方通过键来查询分布式数据库，并且我们数据库应该具备定位拥有该键值对的对等方并返回查询结果的能力。任何对等方也允许在数据库中插入新的键值对。那么这样一种分布式数据库被称为**分布式散列表(Distributed Hash Table, DHT)**。
在P2P文件共享的环境中DHT服务的方式如下：

  • 键是一个目录名，值是拥有该目录副本的对等方IP的地址。
  • 由于DHT数据库分布在一些对等方上，所以某些对等方需要对特定的键负责，并且具有相应的键值对。
  • 假设某个对等方需要获得一个副本，它在下载这个副本之前，必须清楚那个对等方拥有这个副本。为此，用所需副本为键来查询DHT。
  • DHT确定负责这个键的对等方，并从这个对等方那里查询出数据，然后传送回来。

第一章

什么是因特网

具体构成描述

因特网实际上就是互联了所有计算机设备的网络。所有接入网络的设备都称为主机(host)**或着终端(end system)（比较早的时候可能只有传统的桌面PC、服务器等，现在汽车、手表等电子产品都算）。

端系统通过通信链路(communication link)或者分组交换机(packet switch)连接到一起。通信链路实际上就是具体的物理连接方式，比如说电缆、铜线、光纤等。分组交换机主要就是路由器(router)和链路层交换机(link-layer switch)**。

什么是协议

简单来说就是一系列的动作，要求双方或者多方按照这一系列动作来行动，从而完成一项工作。

网络边缘

接入网

**边缘路由器(edge router)**：端系统到任何其他远程端系统的路径上的第一台路由器。

物理媒介

  • 物理媒体(physical medium)
    • 导线型媒体(guided medium)
      • 光缆
      • 双绞线
      • 电缆
    • 非导线型媒体(unguided medium)
      • 陆地无线电信号
      • 卫星无线电信号

        网络核心

        分组交换(packet switching)

报文(message)**：顾名思义，端系统之间相互交换信息。
报文可能较长，不利于传输，由此将报文分成若干个较小的数据块，称为分组(packet)**。分组通过分组交换机（路由器或者链路层交换机）传送。很显然，传送一个L比特的分组，传送速度为R比特每秒，传送时间为L/R秒。

存储转发传输(store-and-forward transmission)

简单来说就是每个分组所有的比特接受完成后，才能向下一个分组交换机发送这个分组，不能边收边发，因此必须将分组缓存下来，接收完成时发送给下一个分组交换机。

排队时延和分组丢失

理想情况下忽略排队时延和分组丢失的情况，然而实际情况并不是这样。
每个分组交换机都有一个输出队列(output queue)**，当输出队列不为空的时候，只有前面的分组被转发过后，当前分组才能被转发，因此就会导致排队时延(queue delay)。同时，输出队列的大小也是一定的，当缓冲队列满后，无法再接受新的分组，导致分组丢失(packet lost)**。

转发表和路由选择协议

分组交换机检查分组中目的地字段，并查**转发表(forwarding table)**，通过路由选择协议(routing protocol)选择合适的下一个分组交换机转发分组。

电路交换(circuit switching)

电话的术语。简单来说就是两个人之间的数据交换必须要保持一个连接的状态，也就是所有进行此次数据交换的资源都被这两个人占用了（有点像操作系统里保持着互斥资源不放），那么这个连接就是一条**电路(circuit)**。

分组交换和电路交换的对比

分组交换批评者的理由：

  1. 延迟不可控。（排队时延以及分组丢失）

分组交换支持者的理由：

  1. 提供更好的带宽。
  2. 比电路交换简单。

分析一下具体的例子：

假设多个用户共享一条1Mbps的链路，每个用户活跃的周期是变化的，时而以100kbps的恒定速率产生数据，时而静止。假设用户有10%的时间活跃，有90%的时间静止（很合理）。那么对于电路交换，需要为每个用户预留100kbps。因此这个电路只能支持10个用户的并发。而通过分组交换，当有十个或者更少的活跃用户时，基本没有延迟。甚至于三倍的用户量时，（35个用户时有11个用户并发使用的概率为大约0.0004），性能几乎相同。

可见分组交换理论上来说是更有效的。

网络的网络

网络结构1

网络结构1很简单，就是在全球范围内建立一个ISP(Internet Service Provider)，那么所有想接入（接入ISP称为**客户(client)**）互联网的人都必须接入这个ISP。

网络结构2

在网络结构1的基础上，多个公司建立了多个全球承载的ISP，那么这些ISP的网络之间必须是可以互通的。否则，只能于同在一个ISP网络下通信。所以在网络结构1的基础上，网络结构2分为居于顶层的多个全球ISP以及基于底层的接入ISP（客户）。

网络结构3

不仅第一层有多个ISP竞争，同时，每个区域也可能有多个ISP在竞争。那么此时，每个接入ISP向他接入的区域ISP付费，而区域ISP又向上层的如国家ISP付费，国家ISP接入第一层ISP。这个结构与现在的互联网结构类似，称为网络结构3。

网络结构4

与现在类似的互联网结构中，存在着一些**存在点(point of presence)**，多宿，对等，和因特网交换点。

  • PoP：一台或多台路由器群组。给接入ISP提供高速链路直接相连。
  • 多宿(multi-home)：任何层的ISP(除了客户)可以选择为多宿，即代表它可以和两个或者多个ISP连接。它可以同时连接上层的ISP和同层的ISP，也可以连接多个上层的ISP。它的上层ISP出问题，它仍可以发送和接受分组。
  • 对等(peer)：位于同一层次的ISP能够对等。对等的ISP之间的流量直接连接而不是通过上层的ISP传输。对等的两个ISP之间流量传输不产生费用付费。
  • 因特网交换点(Internet Exchange Point, IXP):多个ISP在这里对等。

    网络结构5

    它在网络结构4的顶层增加了内容提供商(content provider network)**。内容提供商例如谷歌，有很多数据中心。它的数据中心都是通过专用的TCP/IP相连，并且跨越全球，但独立于公共因特网。它的专用网络通过和较低层的ISP对等**，或者在IPX与他们相连，绕过了较高层的ISP，对他的服务如何交付给端用户有了更多的控制。

    分组交换中的时延、丢包和吞吐量

    分组中的时延概述

    分组从一台主机出发，在另一台主机结束它的历程。分组在沿途中经受的延迟主要有：
  1. 结点处理时延(nodal processing delay)
  2. 排队时延(queuing delay)
  3. 传输时延(transmission delay)
  4. 传播时延(propagation delay)

    时延的类型

  5. 处理时延（数量级为微妙级或更低）
    • 检查分组首部
    • 决定分组到向何处
    • 比特级的差错检测过程
  6. 排队时延（数量级为毫秒到微秒级）
    • 在链路上等待传输
  7. 传输时延（数量级为毫秒到微秒级)
    • 分组从交换机传到链路上的时间
  8. 传播时延(广域网中为毫秒级)
    • 分组在链路上的传播时间

      排队时延和丢包

      排队时延

**流量强度(traffic intensity)**：$La/R$ (L为分组长度，a为分组到达的平均速度，R为传输速度)，设计系统时流量强度不应该大于1。

很显然，$L/R\leq1$的时候，如果分组周期性到达，每$L/R$秒到达一个分组，那么将不会产生排队时延。但是当分组突发性到达，比如一次到达N个分组，那么第一个分组没有排队时延，第二个分组的排队时延为$(L/R)$秒；第N个分组的排队时延为$(L/R)*N$秒。因此可能有很大的排队时延。

丢包

排队的容量极大程度上依赖于路由器设计和成本。所以排队容量时有限的，当流量强队接近1的时候，那么到达的分组将可能发现一个满的队列。由于没有地方存储这个分组，路由器将丢弃(drop)**这个分组，这个分组将会丢失(lost)**。

协议层次及其服务模型

分层的体系结构

协议分层

为了给网络协议的设计提供一个结构，网络设计者以分层(layer)**的方式组织协议以及实现这些协议的网络硬件和软件。我们关注某层向上一层提供的服务(service)。例如，可能第$n$层提供的服务可能包括报文从网络的一边到另一边的可靠传送，这可能是通过第$n-1$层的不可靠报文传送加上第$n$层的检测和重传报文的功能来实现的。
各层所有的协议被称为协议栈(protocol stack)**。因特网的协议栈由五个层次组成：

  1. 物理层
  2. 链路层
  3. 网络层
  4. 运输层
  5. 应用层


ISO制定的OSI(Open System Interconnection，刚好倒过来)模型中多了表示层和会话层。表示层的作用是使得通信的应用程序能够解时交换数据的含义。包括数据压缩和数据加密以及数据描述等服务。会话层提供了数据交换定界和同步功能（感觉没什么用），包括建立了检查点和回复方案的方法。
应用程序的开发者决定是否构建不属于因特网协议栈的功能，如果需要数据加密的功能，那么就在应用程序中实现这个功能。

应用层

处于应用层的信息分组称为**报文(message)**。应用层包括很多协议，主要有：

  1. HTTP：Web文档的请求和传送
  2. SMTP：电子邮件报文的传输
  3. FTP：文件传送
  4. DNS：将对人友好的端系统名字转换为网络地址
    运输层
    运输层的分组称为**报文段(segment)**。运输层在应用程序端点之间传送报文段。因特网中有两个运输协议：
  5. TCP：对应用程序提供了面向连接的服务。服务包括确保传递、流量控制和拥塞控制。
  6. UDP：对应用程序提供无连接服务。没有可靠性，没有流量控制，没有拥塞控制。
    网络层
    网络层的分组称为**数据包(datagram)**。网络层的作用是将分组从一台主机移动到另一台主机。
    因特网的网络层包括著名的IP协议。仅有一个IP协议，所有具有网络层的因特网组件必须运行IP协议。网络层也包括路由选择协议，但是通常简单的称为IP层。
    链路层
    链路层的分组称为**帧(frame)**。链路层向网络层提供将数据报沿着路劲传递给下一个结点的功能。在下个结点，链路层将数据包上传给网络层。
    链路层提供的服务取决于链路层协议。
    物理层
    物理层的任务是将帧中的每一个比特从一个结点移动到下一个结点。

    封装


    应用层报文(application-layer message)被传给运输层，运输层收到报文并附上附加信息。应用层报文和运输层首部信息构成运输层报文段(tranport-layer segment)**。运输层向网络层传递该报文段，网络层又增加了首部信息，产生了网络层数据报(network-layer datagram)。同样的，网络层将数据报向下传给链路层，链路层增加首部信息并创建链路层帧(link-layer frame)。
    所以，我们看到在每一层，一个分组都具有两个类型的字段：首部字段和    有效载荷字段(payload field)**。首部字段是这一层加上的，有效载荷字段是来自上一层的分组。

    面对网络的攻击

    bad guys通过恶意软件(malware)进入并控制我们的设备。于是我们的主机可能称为数以千计类似受害设备网络中的一员，统称为僵尸网络(botnet)。

    拒绝服务攻击(Denial-of-Service (DoS) attack)

    三种类型：
  7. 弱点攻击。向一台目标主机上易受攻击的应用程序或操作系统发送制作精细的报文。
  8. 带宽洪泛。向目标主机发送大量的分组，直接将接入目标主机的链路堵住，使得合法的分组无法到达服务器。
  9. 连接洪泛。在目标主机中创建大量半开或者全开的TCP连接。

DDoS：攻击者通过控制的僵尸网络向目标主机发送海量流量。

一、标题

markdown中用连续的若干个放置在文字之前的’#’来表示标题，一个’#’代表一级标题，两个’#’代表二级标题，以此类推。支持六级标题。

二、字体

加粗

要加粗的文字两边用两个*包起来。需要在同一行才会生效，否则只是普通字符。

如：加粗。

斜体

要倾斜的文字左右分别用一个*号包起来。同样需要在同一行内才会生效。

如：斜体。

斜体加粗

要倾斜和加粗的文字左右分别用三个*号包起来。需要在同一行内才会生效。

如：**斜体加粗**。

删除线

要加删除线的文字左右分别用两个~~号包起来。需要在同一行内才会生效。

如：删除线
